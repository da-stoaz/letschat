# Feature Plan: Space Permissions & Discovery — LetsChat

## Context

A short follow-on to `1-control-panel.md`. Two coupled gaps in the
chat-domain model that the control-panel work didn't reach:

1. **`create_server` has no permission check.** Any signed-in user can call
   it ([`server/src/reducers/servers.rs:7`](../../server/src/reducers/servers.rs))
   and become Owner of an unlimited number of new spaces. For self-hosted
   deployments past the friend-group scale this is open spam surface.
2. **Spaces are invisible to non-members.** The only way to learn a space
   exists is to be invited. That works for tight communities but blocks the
   "browse what's on this instance" experience that public/community
   deployments expect.

This plan adds:

- A **configurable policy** for who may create spaces, runtime-editable from
  the existing admin panel.
- An **opt-in discoverability flag** per space, plus a new
  **"Discover Spaces"** browsing surface in the desktop client.

It does **not** introduce protocol federation or change the existing role
hierarchy — Owner/Moderator/Member stays.

## Sequencing

Plan **1.5**: small follow-on to `1-control-panel.md`, fits between it and
`2-storage-tiering.md`. Independent of plans 2–5; doesn't block any of
them. Builds on plan 1's admin panel and on the existing SpacetimeDB
invite/role infrastructure.

---

## Decisions

| Topic | Decision |
|---|---|
| Where the create-policy is stored | New `SystemSettings` singleton table in **SpacetimeDB** (chat-domain config, read by the reducer at every `create_server` call) |
| How "admin" is recognised by SpacetimeDB | New `is_admin: bool` column on the `User` table. core-api sets it during ASP.NET Identity Admin-role grant/revoke and during the first-login identity rebind of a bootstrap admin |
| Edit surface for the policy | Existing `/admin/config` Razor page gains a "Space settings" card |
| Where discoverability is stored | `is_discoverable: bool` + `description: Option<String>` on the existing `Server` table |
| Edit surface for a space's discoverability | Existing space-management screen (`ServerManagePage` → ServerTab equivalent), Owner-only |
| Join model for discovered spaces | If `invite_policy == Everyone` → one-click join via a new reducer. If `ModeratorsOnly` → display "Ask a moderator to invite you" stub; an actual request-to-join workflow is **out of scope** for this plan |
| Default discoverability | `false`. Operator/owner has to opt in per space |
| Default create policy | `Anyone` (preserves existing behaviour). Operator flips it via `/admin/config` if they want to lock down |
| Anti-spam (description length, max discoverable per owner) | Description capped at 280 chars at the reducer. Per-owner discoverable quota deferred — flag in code as a comment, revisit if abuse appears |

## Phases

### Phase 1 — Space creation policy

**SpacetimeDB schema additions** ([`server/src/schema.rs`](../../server/src/schema.rs)):

- `SpaceCreatePolicy` enum: `Anyone | AdminsOnly`.
- `SystemSettings` singleton table (single row, fixed id):
  - `space_create_policy: SpaceCreatePolicy` (default `Anyone`)
  - room for future flags without further migrations
- `User.is_admin: bool` (default `false`).

**SpacetimeDB reducers** (new `server/src/reducers/system.rs`):

- `set_space_create_policy(policy: SpaceCreatePolicy)` — gated to callers
  whose `User.is_admin = true`.
- `set_user_admin(target: Identity, is_admin: bool)` — same gate; used by
  core-api to push role changes through.

**Existing `create_server` reducer** ([`server/src/reducers/servers.rs:7`](../../server/src/reducers/servers.rs)):
read the singleton, branch on policy; reject with a clear error if
`AdminsOnly` and caller isn't an admin.

**core-api integration**:
- On `UserManager.AddToRoleAsync(user, "Admin")` / `RemoveFromRoleAsync` for
  the system Admin role, call the SpacetimeDB `set_user_admin` reducer with
  the affected user's SpacetimeIdentity. Needs a SpacetimeDB identity for
  core-api itself — see "Bootstrap & identity for core-api" below.
- During Login identity-rebind for a bootstrap-admin account, also push
  `set_user_admin(true)` so the first admin gets the SpacetimeDB-side flag
  without a separate ceremony.

**Admin panel** ([`core-api/src/CoreApi/Pages/Admin/Config.cshtml`](../../core-api/src/CoreApi/Pages/Admin/Config.cshtml)):
new "Space settings" card with a radio: `Anyone (default) / Admins only`.
Edits call a new core-api endpoint `/admin/spaces/policy` which invokes the
SpacetimeDB reducer.

### Phase 2 — Discoverability + Discover panel

**SpacetimeDB schema additions**:

- `Server.is_discoverable: bool` (default `false`)
- `Server.description: Option<String>` (max 280 chars at the reducer)

**SpacetimeDB reducers**:

- `set_server_discovery(server_id, is_discoverable, description)` — Owner-only
  on the target server.
- `join_discoverable_server(server_id)` — caller becomes a Member, with these
  checks: server exists, `is_discoverable = true`, `invite_policy = Everyone`,
  caller isn't already a member, caller isn't banned.

**Desktop client**:

- New store: [`src/stores/discoverStore.ts`](../../src/stores/discoverStore.ts) —
  syncs the public `Server` rows where `is_discoverable = true` and the caller
  isn't a member.
- New page: [`src/pages/DiscoverPage.tsx`](../../src/pages/DiscoverPage.tsx) —
  list/grid of cards per discoverable space, each showing icon, name,
  description, member count, and an invite-policy badge.
- New entry in [`src/layouts/app-layout/AppRail.tsx`](../../src/layouts/app-layout/AppRail.tsx) —
  "Discover" rail item (compass icon), alongside DM Home and Settings.
- Per card:
  - `invite_policy = Everyone` → primary "Join" button calling
    `join_discoverable_server`.
  - `invite_policy = ModeratorsOnly` → disabled button with tooltip
    "Ask a moderator to invite you."
- Existing `ServerTab` ([`src/features/server-manage/ServerTab.tsx`](../../src/features/server-manage/ServerTab.tsx))
  gains a "Discovery" section: toggle for `is_discoverable` and a
  280-char description textarea. Owner-only.

### Phase 3 — Optional polish (defer unless validated by use)

- Topic tags on Server, filterable in Discover.
- Free-text search across name/description.
- "Request to join" workflow for `ModeratorsOnly` discoverable spaces:
  notification to moderators, accept/decline UI, audit trail.
- Per-owner discoverable-space quota for anti-spam.
- Admin-of-instance ability to **un-list** a space they don't own (community
  moderation). Currently only the space Owner can unset `is_discoverable`.

## Bootstrap & identity for core-api

To push role-grant events from core-api into SpacetimeDB, core-api needs
a SpacetimeDB identity it can use to call reducers. Three options:

1. **Module owner key.** SpacetimeDB modules have a publisher key — calls
   made with it are unconditionally trusted. core-api would carry that key
   and call reducers directly. Tightest coupling between core-api and the
   module; smallest surface.
2. **Dedicated `system` SpacetimeDB user** whose Identity is held by
   core-api and bootstrapped at publish time. `is_admin = true`. core-api
   signs reducer calls as this user.
3. **Reducer-side allowlist** (e.g., `SystemSettings.admin_identities:
   Vec<Identity>`) seeded with one known core-api identity at publish.

Decision deferred until implementation; (2) is the most explicit and
keeps the trust boundary visible in the schema. (1) is simpler if the
module-owner key story is straightforward to expose to core-api.

## Verification Checklist

1. Policy `Anyone` (default): any active user can create spaces — current
   behaviour preserved.
2. Policy `AdminsOnly`: a non-admin's `create_server` call fails with a
   clear permission error; an admin's call succeeds and they become Owner.
3. Granting Admin role to a user in core-api propagates `User.is_admin = true`
   in SpacetimeDB; revoking flips it back to `false`.
4. Bootstrap admin gets `is_admin = true` on first identity rebind (no
   second ceremony required).
5. Marking a space `is_discoverable = true` with a description shows it on
   the Discover panel within sync latency.
6. Discover panel hides spaces the caller is already a member of and ones
   that aren't discoverable.
7. Joining a discoverable Everyone-policy space via the panel creates a
   ServerMember row; the panel then hides that space from the joiner.
8. Joining a discoverable ModeratorsOnly-policy space via the panel is
   disabled with the correct tooltip.
9. Description over 280 chars is rejected at the reducer; the textarea
   prevents it client-side.
10. `/admin/config` policy change persists and the next `create_server`
    call reflects the new policy without restart.
11. Banned users can't join via the Discover flow (regression check on
    the existing ban table).

## Effort

~1–2 weeks single developer.

- Phase 1: ~3–4 days. Most of it is the core-api ↔ SpacetimeDB identity
  story (decision + plumbing), not the reducer or admin-panel surfaces.
- Phase 2: ~3–4 days. Schema additions, two reducers, one page, one
  store, one rail entry, one ServerTab section.
- Phase 3: deferred.
